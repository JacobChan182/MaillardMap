# BigBack Architecture Specification

## 1. PostgreSQL Schema

All tables use `uuid` primary keys generated by `gen_random_uuid()` from the `pgcrypto` extension. Timestamps are `timestamptz` with `default now()`.

### 1.1 `users`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `uuid` | `PRIMARY KEY DEFAULT gen_random_uuid()` | User identifier |
| `username` | `text` | `NOT NULL UNIQUE` | Public handle, 3-32 chars |
| `password_hash` | `text` | `NOT NULL` | bcrypt hash (cost 12) |
| `phone` | `text` | `NULLABLE` | Phone number (optional, for future SMS) |
| `email` | `text` | `NULLABLE` | Email address (optional) |
| `created_at` | `timestamptz` | `NOT NULL DEFAULT now()` | Account creation time |

**Indexes:**
- `users_username_idx`: `UNIQUE` on `username` (implicit via `UNIQUE` constraint)

### 1.2 `friendships`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `uuid` | `PRIMARY KEY DEFAULT gen_random_uuid()` | Friendship record ID |
| `user_id` | `uuid` | `NOT NULL REFERENCES users(id) ON DELETE CASCADE` | Requesting user |
| `friend_id` | `uuid` | `NOT NULL REFERENCES users(id) ON DELETE CASCADE` | Recipient user |
| `status` | `text` | `NOT NULL CHECK (status IN ('pending', 'accepted'))` | Friendship status |
| `created_at` | `timestamptz` | `NOT NULL DEFAULT now()` | Request creation time |

**Indexes:**
- `friendships_user_idx`: `btree` on `(user_id, status)`
- `friendships_friend_idx`: `btree` on `(friend_id, status)`
- `friendships_pair_idx`: `UNIQUE btree` on `(LEAST(user_id, friend_id), GREATEST(user_id, friend_id))` (prevents duplicate pairs)

**Constraints:**
- `friendships_no_self`: `CHECK (user_id <> friend_id)`

### 1.3 `restaurants`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `uuid` | `PRIMARY KEY DEFAULT gen_random_uuid()` | Internal identifier |
| `foursquare_id` | `text` | `NOT NULL UNIQUE` | Foursquare Places API ID |
| `name` | `text` | `NOT NULL` | Restaurant name |
| `address` | `text` | `NULLABLE` | Formatted address from Foursquare |
| `lat` | `double precision` | `NOT NULL` | Latitude |
| `lng` | `double precision` | `NOT NULL` | Longitude |
| `cuisines` | `text[]` | `NOT NULL DEFAULT '{}'` | Array of cuisine categories |
| `fetched_at` | `timestamptz` | `NOT NULL DEFAULT now()` | Last sync from Foursquare |

**Indexes:**
- `restaurants_foursquare_idx`: `UNIQUE` on `foursquare_id` (implicit via `UNIQUE` constraint)
- `restaurants_location_idx`: `btree` on `(lng, lat)` (for proximity searches)
- `restaurants_cuisines_idx`: `GIN` on `cuisines` (for cuisine overlap queries)

### 1.4 `posts`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `uuid` | `PRIMARY KEY DEFAULT gen_random_uuid()` | Post identifier |
| `user_id` | `uuid` | `NOT NULL REFERENCES users(id) ON DELETE CASCADE` | Author |
| `restaurant_id` | `uuid` | `NOT NULL REFERENCES restaurants(id) ON DELETE CASCADE` | Which restaurant |
| `comment` | `text` | `NOT NULL CHECK (length(comment) <= 200)` | Short comment, max 200 chars |
| `created_at` | `timestamptz` | `NOT NULL DEFAULT now()` | Post time |

**Indexes:**
- `posts_user_idx`: `btree` on `(user_id, created_at DESC)`
- `posts_restaurant_idx`: `btree` on `(restaurant_id, created_at DESC)`
- `posts_feed_idx`: `btree` on `created_at DESC` (for feed ordering)

### 1.5 `post_photos`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `uuid` | `PRIMARY KEY DEFAULT gen_random_uuid()` | Photo identifier |
| `post_id` | `uuid` | `NOT NULL REFERENCES posts(id) ON DELETE CASCADE` | Parent post |
| `url` | `text` | `NOT NULL` | S3 object URL (pre-signed or CDN) |
| `order_index` | `integer` | `NOT NULL CHECK (order_index BETWEEN 1 AND 3)` | Display order (1-3) |

**Indexes:**
- `post_photos_post_idx`: `btree` on `(post_id, order_index)`

**Constraints:**
- Max 3 photos per post enforced by service layer + DB `CHECK` on `order_index`
- Unique constraint on `(post_id, order_index)` to prevent duplicates

### 1.6 `likes`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `uuid` | `PRIMARY KEY DEFAULT gen_random_uuid()` | Like identifier |
| `user_id` | `uuid` | `NOT NULL REFERENCES users(id) ON DELETE CASCADE` | Who liked |
| `post_id` | `uuid` | `NOT NULL REFERENCES posts(id) ON DELETE CASCADE` | Which post |
| `created_at` | `timestamptz` | `NOT NULL DEFAULT now()` | Like time |

**Indexes:**
- `likes_unique_idx`: `UNIQUE btree` on `(user_id, post_id)` (one like per user+post)
- `likes_post_idx`: `btree` on `post_id` (for counting likes per post)

### 1.7 `saved_places`

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `uuid` | `PRIMARY KEY DEFAULT gen_random_uuid()` | Save record ID |
| `user_id` | `uuid` | `NOT NULL REFERENCES users(id) ON DELETE CASCADE` | Who saved |
| `restaurant_id` | `uuid` | `NOT NULL REFERENCES restaurants(id) ON DELETE CASCADE` | Which restaurant |
| `created_at` | `timestamptz` | `NOT NULL DEFAULT now()` | Save time |

**Indexes:**
- `saved_places_unique_idx`: `UNIQUE btree` on `(user_id, restaurant_id)`
- `saved_places_user_idx`: `btree` on `(user_id, created_at DESC)`

### 1.8 `schema_migrations` (system)

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | `text` | `PRIMARY KEY` | Migration filename (e.g., `001_create_users.sql`) |
| `applied_at` | `timestamptz` | `NOT NULL DEFAULT now()` | When migration was applied |

## 3. Foursquare Places API Integration Plan

### 3.1 Overview

Foursquare is the single source of truth for all restaurant data. Restaurants are **never** created directly by users. The backend proxies Foursquare and caches results in the local `restaurants` table.

### 3.2 Endpoints Used

| Foursquare Endpoint | Purpose | Method |
|---------------------|---------|--------|
| `GET https://api.foursquare.com/v3/places/search` | Search restaurants by query + location | Proxied via `/restaurants/search` |
| `GET https://api.foursquare.com/v3/places/{foursquare_id}` | Get details for a single place | On-demand when creating posts |
| `GET https://api.foursquare.com/v3/places/categories` | Map category IDs to cuisine labels (cached statically) | Bootstrap |

### 3.3 Caching Strategy

| Layer | Mechanism | TTL |
|-------|-----------|-----|
| DB `restaurants` table | Store fetched Foursquare data in local `restaurants` table | 24 hours (check `fetched_at`) |
| Search results (in-memory) | LRU cache keyed by `query + lat + lng` | 5 minutes |

When a search query arrives:
1. Check `restaurants` table for exact Foursquare matches matching the query
2. Check in-memory search cache
3. If neither hits, call Foursquare API
4. Cache results and upsert into `restaurants` table

### 3.4 Restaurant Sync Flow

1. User searches restaurants via mobile app
2. App calls `GET /restaurants/search?q=...&lat=...&lng=...`
3. Backend calls Foursquare Places Search API
4. For each result, upsert into `restaurants` table (ON CONFLICT update `fetched_at` and mutable fields)
5. Return results to client

When creating a post:
1. Client passes a restaurant `id` (internal UUID) that was previously fetched via search
2. If the restaurant somehow isn't in the DB, fetch details from Foursquare by `foursquare_id` and insert
3. Create the post

### 3.5 Rate Limiting

- Foursquare API has its own rate limits. The backend implements a per-user rate limiter on `/restaurants/search`: max 20 requests per minute.
- Cached responses reduce Foursquare API calls significantly.
- The recommendation system also calls Foursquare but only once per blend request, not per user.

### 3.6 Data Mapping

| Foursquare Field | Backend Field | Notes |
|-----------------|---------------|-------|
| `fsq_id` | `foursquare_id` | Primary key mapping |
| `name` | `name` | Direct |
| `geocodes.main.latitude` | `lat` | Direct |
| `geocodes.main.longitude` | `lng` | Direct |
| `categories[].name` | `cuisines` | Collect all category names; deduplicate |
| `location.formatted_address` | `address` | Optional |

## 4. Mapbox Integration Plan

### 4.1 Overview

Mapbox provides map rendering, tiles, and visualization for both iOS and Android apps. The mobile apps use Mapbox SDKs directly; the backend does not proxy Mapbox.

### 4.2 Map SDK Versions

| Platform | SDK | Minimum Version |
|----------|-----|-----------------|
| iOS | Mapbox Maps SDK for Swift | v11.x |
| Android | Mapbox Maps SDK for Android / Kotlin | v11.x |

### 4.3 Map Layers

#### Layer 1: Base Map
- Mapbox Streets or Mapbox Light style
- Default zoom: street-level (14-16)

#### Layer 2: Heatmap (Zoom < 12)
- Source: GeoJSON cluster of post coordinates
- Density visualization showing friend activity concentration
- Data source: `GET /posts/feed` or a dedicated `GET /map/posts?bbox=...` endpoint
- Implementation: Mapbox Expression with `heatmap` layer type
- Color ramp: Light (blue) -> Dark (red) for activity density

#### Layer 3: Pins (Zoom >= 12)
- Individual pins for:
  - Restaurants where friends posted
  - Saved places (bookmarked icon)
  - Recommended blend results (star icon)
- Pin clustering enabled at medium zoom levels (12-14)
- Tap a pin -> show restaurant name + friend activity

### 4.4 Data Flow

1. App centers map on user's current location (from device GPS)
2. App fetches posts/restaurants for the current viewport bounding box
3. Backend endpoint `GET /map/posts` (to be implemented) returns GeoJSON or lightweight JSON:
   ```json
   {
     "features": [
       {
         "restaurant_id": "uuid",
         "lat": 40.758,
         "lng": -73.9855,
         "name": "Restaurant Name",
         "post_count": 3,
         "friends": ["user1", "user2"]
       }
     ]
   }
   ```
4. Client converts to GeoJSON and renders with Mapbox layers
5. Zoom threshold toggles between heatmap layer and pin layer at zoom level 12

### 4.5 Authentication

- Mapbox access token is embedded in mobile apps (public token only)
- Token should be restricted to the app's bundle ID/package name in Mapbox dashboard
- No Mapbox API calls go through the backend

## 5. Heuristic Recommendation Algorithm (NO ML)

### 5.1 Input

- `user_ids`: Array of user UUIDs to blend (includes the requesting user implicitly)

### 5.2 Step-by-Step Algorithm

```
FUNCTION blend(user_ids):

  STEP 1: COLLECT SIGNALS
    -- Restaurants from all selected users' posts
    restaurants_from_posts = query(
      SELECT DISTINCT p.restaurant_id
      FROM posts p WHERE p.user_id = ANY(user_ids)
    )

    -- Restaurants from all selected users' liked posts
    restaurants_from_likes = query(
      SELECT DISTINCT p.restaurant_id
      FROM likes l JOIN posts p ON l.post_id = p.id
      WHERE l.user_id = ANY(user_ids)
    )

    -- Restaurants from all selected users' saved places
    restaurants_from_saved = query(
      SELECT DISTINCT s.restaurant_id
      FROM saved_places s WHERE s.user_id = ANY(user_ids)
    )

    all_restaurant_ids = UNION(restaurants_from_posts, restaurants_from_likes, restaurants_from_saved)

    IF all_restaurant_ids IS EMPTY:
      RETURN error "INSUFFICIENT_DATA"

  STEP 2: COMPUTE CUISINE FREQUENCIES
    cuisine_counts = query(
      SELECT unnest(cuisines) AS cuisine
      FROM restaurants WHERE id = ANY(all_restaurant_ids)
      GROUP BY cuisine ORDER BY COUNT(*) DESC
    )

    top_cuisines = top 5 cuisines from cuisine_counts

  STEP 3: COMPUTE GEOGRAPHIC CENTROID
    centroid = query(
      SELECT avg(lat) AS avg_lat, avg(lng) AS avg_lng
      FROM restaurants WHERE id = ANY(all_restaurant_ids)
    )

  STEP 4: QUERY NEARBY RESTAURANTS
    nearby = foursquare.search(
      near = {lat: centroid.avg_lat, lng: centroid.avg_lng},
      categories = top_cuisines,
      limit = 50,
      radius = 10km
    )

    upsert all results into restaurants table

  STEP 5: RANK RESULTS
    FOR EACH restaurant r in nearby:
      r.cuisine_match_count = COUNT of r.cuisines that ARE IN top_cuisines
      r.distance_km = haversine_distance(
        {lat: r.lat, lng: r.lng},
        {lat: centroid.avg_lat, lng: centroid.avg_lng}
      )

      -- Score: combine cuisine match strength with proximity
      -- Higher cuisine overlap = better, closer = better
      cuisine_score = r.cuisine_match_count / top_cuisines.length
      distance_score = 1.0 - min(r.distance_km / 10.0, 1.0)

      r.score = 0.7 * cuisine_score + 0.3 * distance_score

    SORT nearby BY r.score DESC

  STEP 6: RETURN
    RETURN {
      top_cuisines: cuisine_counts[:5],
      centroid: centroid,
      recommendations: top 20 from nearby
    }
```

### 5.3 Weight Rationale

| Factor | Weight | Why |
|--------|--------|-----|
| Cuisine match score | 70% | Core "taste blending" signal -- users care most about food type |
| Distance score | 30% | Proximity matters but secondary to taste preferences |

### 5.4 Performance Considerations

- Step 1-3: Single SQL query each with `ANY()` arrays. Uses indexes. O(number of selected users).
- Step 4: External API call (Foursquare). This is the slowest step. Cache results in `restaurants` table. Use TTL-based fetch.
- Step 5: In-memory computation over ≤50 results. Negligible.
- Target response time: <500ms total (excluding Foursquare latency, which is variable).

### 5.5 Strict Prohibitions

- NO machine learning models
- NO embeddings or vector similarity
- NO collaborative filtering
- NO user behavior tracking beyond what they explicitly do (post, like, save)
- Scoring is purely arithmetic (frequency counts + haversine distance)

| **Password hashing** | bcrypt with cost factor 12 |
| **JWT expiration** | 7 days, signed with HS256 |
| **JWT_SECRET minimum** | 32 characters, validated at startup |
| **Rate limiting** | express-rate-limit: 300 requests / 60s window (global). Auth endpoints: stricter limit recommended (10/min) |
| **CORS** | Enabled for mobile client origins |
| **Helmet headers** | XSS protection, HSTS, frame guards via `helmet` middleware |
| **Input validation** | Zod schemas for all request bodies |
| **SQL injection prevention** | Parameterized queries via `pg` driver; no raw string concatenation |
| **Photo uploads** | S3 pre-signed URLs only; no direct file upload to backend; file type restricted to jpg/png; max file size enforced |
| **Comments** | Max 200 chars validated in Zod schema + DB CHECK constraint |
| **Photos per post** | Max 3, validated in service layer |
| **Friendship validation** | Prevent self-friendship via DB CHECK; prevent duplicate pairs via composite unique index |
| **Authorization** | JWT `sub` claim used to verify resource ownership. Users can only access their own data and friends' public data |
| **Secrets** | Never committed to git; use `.env` (in `.gitignore`); `.env.example` provided with placeholder values |

### 6.2 Performance

| Rule | Target | How |
|------|--------|-----|
| **Endpoint latency** | <200ms (p50), <500ms (p95) for non-external-API endpoints | Connection pool size 10, simple queries, no N+1 |
| **N+1 prevention** | Use `JOIN` or `IN ()` arrays, never one query per item in loop | `posts.feed` fetches all author + restaurant data in one query |
| **Foursquare caching** | DB cache: 24h TTL; in-memory LRU: 5 min TTL | Avoid redundant external calls |
| **Feed pagination** | Cursor-based (`before` timestamp), not offset | Consistent latency regardless of feed size |
| **Photo delivery** | CDN-backed S3 with pre-signed URLs with 10-minute expiry | No backend proxy for image delivery |
| **No UI blocking** | Mobile apps must use async calls with loading states | Never block the main thread |
| **Connection pooling** | Max 10 connections to Postgres | Suitable for single-node deployment |
| **No premature abstractions** | Simple queries over ORMs; no over-generic services | Keep code readable and fast |
| **Build size** | Keep backend dependency tree minimal | Currently: express, pg, zod, jsonwebtoken, bcryptjs, helmet, cors, morgan, express-rate-limit |

### 6.3 Guardrails (DO NOT VIOLATE)

1. **NEVER** allow custom restaurant creation -- restaurants only come from Foursquare
2. **ALWAYS** use Foursquare IDs as the external identifier
3. **NEVER** introduce realtime systems (no WebSockets, no Server-Sent Events, no long polling)
4. **NEVER** exceed 3 photos per post
5. **NEVER** exceed 200 characters per comment
6. **NEVER** use ML for recommendations
7. **NEVER** block the UI on slow network calls (mobile clients must handle loading/error states)
